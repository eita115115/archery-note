# Archery Note デザイン言語 — "Field Instrument"

UI 全面改修（2026-07 開始）の正本。以降の UI PR はすべてこの文書に従う。
違反はレビューで差し戻す。ここに無い判断が必要になったら実装せず設計者（ハブ）へ返す。

## コンセプト

**射場に持ち込む精密な道具。** アーチェリーの用具（カーボン、金属、スコアカードの紙）の
質感と規律を持ち、Web ページではなく「計測機器＋記録帳」に見えること。

判断基準は常に2つ:

1. **射場で使えるか** — 屋外の直射日光、手袋、片手、疲労時。迷いなく読めて押せるか
2. **プロが作ったように見えるか** — 不揃い・例外・装飾のための装飾が1つもないか

## 1. 色 — カーボン＋ゴールド

- **ベースは無彩色**: light = 温かみのある紙白＋墨、dark = カーボングラファイト。
  グラデーション背景・ガラス風は廃止
- **ブランドアクセントはゴールド（的の10点色系）1色**: 主 CTA・選択状態・現在地。
  現行の緑はアクセントから退役し「成功/完了の状態色」のみに降格
- **的の色（金/赤/青/黒）はデータ専用色**: 得点・リング由来の表示にのみ使う。
  装飾に使ったら違反
- 1画面内の有彩色は「アクセント1＋データ色」まで。新しい色をトークン外で作らない

## 2. タイポグラフィ — 数字が主役

- フォントはシステムスタック維持（オフライン・審査対応）。ただし規律で差を出す:
  - **数値はすべて `font-variant-numeric: tabular-nums`**（桁揃え・計器感）
  - 主役数値（合計点・平均・残り本数）は専用スタイル（大サイズ＋重ウェイト＋タイト行間）
  - 見出しは小さく・ラベル的に（セクション名は小サイズ＋letter-spacing＋淡色 = 刻印風）
- 1カード内のフォントサイズは最大2段階。本文はトークンの type scale のみ使用

## 3. 形と面 — 目盛りと罫線

- **角丸を縮小**: pill（999px）はチップのみ。ボタン/カード/パネルは 8–12px（精密機器の面取り）
- **「白カード＋薄ボーダー＋影」のダッシュボード構成を廃止**。代わりに:
  - セクションは**罫線（hairline）と余白**で区切る（スコアカードの紙の文法）
  - 面の階層は最大2つ: 地（背景）と、浮かせる必要がある面（モーダル・アクティブHUD）のみ影を持つ
- select は `appearance:none`＋自前の矢印。`confirm()` は廃止し共通ダイアログ（openModal 系）に統一。
  スクロールバーはデスクトップで細く控えめにスタイルする
- 情報密度は**シチュエーションで変える**: 記録系画面=疎・巨大・即読、分析系画面=密でよいが節ごとに呼吸

## 4. モーション — 目的のない動きは書かない

全アニメーションは以下4目的のどれかを CSS コメントで宣言する（`/* motion:因果 */` の形式）。
宣言できない動きは実装しない。

| 目的 | 用途                                 | 例                                                     |
| ---- | ------------------------------------ | ------------------------------------------------------ |
| 因果 | 操作の結果がどこに反映されたか見せる | 的タップ→得点チップが的の方向から現れる                |
| 方向 | 空間の移動を示す                     | ステージ遷移・タブ切替のスライド方向は実際の順序に一致 |
| 状態 | 値の変化を知覚させる                 | HUD の残り本数・合計のティック更新                     |
| 注意 | 不可逆操作の直前のみ                 | 破壊的ダイアログの登場                                 |

- 時間・イージングは既存トークン（--motion-fast/med/fluid、--ease-app/fluid）のみ
- `prefers-reduced-motion` 対応を維持。check-ui の否定 assert（targetImpact 等の復活禁止）に抵触しない

## 5. シチュエーション設計 — 2つのモード

### 射場モード（記録・記録中）

- 親指到達圏（画面下半分）に全操作。的は最大サイズ
- 1本記録=1タップ・エンド確定=1タップの現行導線を絶対に悪化させない
- 表示は「いま必要な3つ」まで: 的・直近エンド・残り。それ以外は畳む
- コントラストは WCAG AA を最低線に、主要素は AAA 目標（直射日光）

### 机上モード（分析・履歴・サイト・用具）

- 読み物として設計。上から「結論→根拠→詳細」の順
- **初心者文法（分析カードの必須構成）**:
  1. 平易な見出し（専門語を使わない）
  2. 主数値（大きく・単位付き）
  3. **一言解釈**（「◯◯なので△△すると良い」形式の自動生成文。数値の意味を日本語で言い切る）
  4. 「詳しく」（details 折りたたみに専門値・計算根拠）
- 専門語を出す場合は必ず平易語を併記（例:「グルーピング（矢の集まり具合）」）

## 6. プロ品質チェックリスト（各 PR のレビュー項目）

- [ ] トークン外の色・サイズ・角丸・時間が diff に無い
- [ ] 同種要素の間隔・整列が全箇所一致（±0px）
- [ ] 数値が tabular-nums で桁揃えしている
- [ ] 押下・フォーカス・無効・空状態がすべて定義されている
- [ ] ダークモードで再検証済み（スクリーンショット添付）
- [ ] アニメーションに目的コメントがある
- [ ] 3ビューポート（390/360/1280）で崩れなし
- [ ] 分析系カードは初心者文法（見出し/主数値/一言解釈/詳しく）に従う

## 7. 工程の掟

- **1 PR = 1画面（または基盤1件）**。見た目の変更とロジックの変更を混ぜない
- storage schema・保存形式・SW・backup/import/export・採点/記録ロジックは触らない
- check-ui / check-app の文言 assert と e2e は**同一 PR で同期更新**。
  改修した画面の主要素には `data-testid` を導入し、e2e を文言依存から移行していく
- PR 順序: P0 基盤（トークン v2＋プリミティブ＋クローム）→ P1 記録 → P2 分析 → P3 履歴 → P4 サイト → P5 用具・設定 → P6 磨き
