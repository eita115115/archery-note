## 0. 運用モデル（Fable なしでどう回すか）

役割は人ではなく**プロンプトの型**。すべて `Agent(model:'sonnet')` で実行できる。

1. **設計/分解**: `.claude/agents/an-planner.md` の型。要望を1タスク=1関心事に割る。
2. **実装**: `an-implementer.md` の型。1 worktree・1タスク・push しない。
3. **独立検証**: `an-verifier.md` の型。報告を信じず git 実体と再実行で確認。
4. **厳格レビュー**: `an-reviewer.md` の型。Critical/Major を潰す。
5. **監督の代行**: 「**docs/ui-design-language.md（v2）の7節チェックリストと本 roadmap の決定に照らして監査せよ**」と Sonnet に指示すれば、Fable の裁定の大半は再現できる。スクリーンショットは `Read` で画像を直接見て判定する（このプロジェクトの監督はスクショ目視が要）。

**マージの定型手順（1チェーンで書く。CI をすり抜けない）:**

```
rebase origin/main → check:all/lint/format/e2e → push → PR 作成
→ チェック登録待ちループ → gh pr checks --watch → exit 行を case 判定
→ 0 なら merge、非0 なら中身を見る（check:ui flake は rerun、それ以外は調査）
```

- **CI 失敗時に無条件マージしない。** exit 行を必ず読む。
- **check:ui は起動フレークが出る**（v1.6.1 で堅牢化予定。まだ出たら `gh run rerun --failed`）。
- **公開後は必ず `version.json` の実配信を確認**。Pages デプロイは一時失敗が頻発するので `gh run rerun` で復旧。
- version マーカーは4箇所（APP_VER / version.json / sw.js キャッシュ名 / package.json）を `npm run version:bump` で一括。**tools/ と docs/ だけの変更は bump しない**（配信物に影響しない）。

**並行セッション注意:** 別セッションが同リポで走ることがある（例: PR #113 は別作業）。必ず origin/main へ rebase してから PR。worktree で分離。

## 2. 決定済みの判断（再確認不要）

### D-1. 射形トラッキングの評価軸 = 「自分基準」で確定

エリート比較採点（`FORM_REF`・`formGaussScore`）は妥当性監査（判定C）で**出典追跡不能＋カメラ角度誤差が許容幅を飲む**ため v1.6.0 で表示停止・コード内で未使用化。
**決定: 自分中央値比較を恒久的な評価軸とする。** エリート採点の復活は「検証可能な一次データセット（複数人・複数条件）」が入手できた場合に**限り**再検討。それまで復活提案は却下でよい。

### D-2. 矢プレゼンス検出（arrowCheck）の昇格基準 — ★次の主要マイルストーン

現状: v1.6.1 でシャドー（注釈のみ、取消しない）。分離性は合成で完璧（あり≥0.774 / なし0.0 / 閾値0.55）。
**昇格（自動取消への昇格）のゲート = ユーザーの実射検証で以下を全て満たすこと:**

- (G1) 通常の射 ≥20 本で「矢: 発射と一致」が付き、`letdown-mismatch` の誤付与が 0。
- (G2) 意図的レットダウン数回で、（そもそも射としてカウントされないのが基本だが）もし誤カウントされた場合に `letdown-mismatch` が付く。
- (G3) 撮影中の FPS/HUD にカクつきが出ない。
  **昇格の実装（ゲート通過後）:** 速度スパイク発火後の確定猶予窓（`CONFIRM_MS`）で `judgeArrowCheck` が `letdown-mismatch` かつ矢在スコアが閾値以上なら、**既存の `canceled` 自己修復パスを発火**させて射を取り消す。低コントラストで unclear のときは**従来通り取り消さない**（誤取消より誤カウントの方が傷が浅い＝安全側）。手動削除ボタンは残す。
  **この昇格で D-3 の 50ms 問題も実質クローズする**（矢を直接見るので速度盲点を補える）。

### D-3. 「50ms 単一フレームのレットダウン」限界 = 保留で正しい

速度だけでは実リリースと区別不能。**arrowCheck 昇格（D-2）が本命の解**。単独での閾値いじりは誤検出を増やすので**やらない**。

### D-4. リリース判定方式 = 瞬間速度スパイク単独で確定

v1.6.0 で `RELEASE_TH` を body-lengths/sec 単位に再チューン（1.2→9）。累積離脱量方式には戻さない（レットダウン誤検出の元凶だった）。`check-form-core.js` の境界スイープ回帰テストを壊さないこと。

### D-5. リリース手順・CI・Pages = 0節の定型で確定

これ以上プロセス判断を仰がない。
